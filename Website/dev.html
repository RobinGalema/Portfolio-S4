Gebruik van SASS

V

Ontwikkelomgeving

Inleiding
Voor het maken van deze portfoliowebsite wilde ik mezelf aan gaan leren om SASS te gaan gebruiken in plaats van normaal CSS. Ik heb hierover namelijk vorig semester een paar keer over gelezen en het leek mijeen leuk idee om te experimenteren met SASS en de functies hiervan. Aangezien ik mijn portfoliowebsite zelf wilde maken en geen template wilde gebruiken leek mij dit een prima gelegenheid om SASS te gaan gebruiken. Hoe ik hiermee werk zal onder dit kopje beschreven staan.

Orientatie & Eerste gebruik
Omdat ik over het algemeen nog niet veel wist over de functies van SASS en alle mogelijkheden van SASS ben ik begonnen met het kijken van een Crash Course SASS op YouTube. In deze video wordt in 45 minuten een responsive webpagina gebouwd met behulp van SASS.

Na het kijken van deze video ben ik begonnen met het maken van mijn portfoliowebsite. Over de opzet hiervan staat meer beschreven bij het kopje "Opbouw Portfoliowebsite". Ik heb voor mijn IDE (Visual Studio Code) een Live SASS Compiler geinstalleerd die SASS uit mijn .SCSS bestand automatisch omzet naar CSS zodat mijn site in elke browser goed weergegeven wordt. Vervolgens ben ik begonnen met het bouwen van mijn landing-pagina en heb ik de hele tijd in mijn .SCSS bestand gewerkt en geprobeerd, waar het mogelijk was, om de functionaliteit van SASS te gebruiken.

Gebruik tot nu toe
Tot nu toe gebruik ik een aantal handigheidjes die SASS biedt tegenover normaal CSS. In de onderstaande code-snippet is te lezen hoe ik een variable in SASS gebruik om al mijn verschillende kleurcodes op te slaan en deze vervolgens met een functie toewijs.

Bovenin mijn .SCSS bestand declareer ik alle kleuren die ik op mijn pagina ga gebruiken met een variabele. Vervolgens maak ik een functie die de map-get functie van css uitvoert en automatisch een van de 2 parameters invult, hierdoor hoef je alleen nog maar de kleur aan te geven die je wilt gebruiken. Hierdoor hoef ik alleen maar de naam van de kleur te typen en niet bij elk element opnieuw de rgba code invullen. Ook is het zo makkelijker om een aanpassing in het kleurenschema te maken omdat ik maar op een plaats de rgba waarde hoef aan te passen en vervolgens alle elementen met deze kleur automatisch geupdate worden.

Naast deze variable gebruik ik ook nesting van bijvoorbeeld de :hover. In SCSS hoef ik deze niet apart in een nieuw blok te zetten maar kan ik deze nesten in de css van het element zelf zoals in het voorbeeld hieronder.

In dit stukje SCSS staan p en a als child van courseContent, iets wat met normaal CSS ook mogelijk is. Bij de a staat nu echter ook de &:hover en de &:visited meteen in hetzelfde blok waar deze normaal apart moeten worden opgescreven. Op deze manier heb ik een beter overzicht van al mijn events en kan ik in een keer zien bij welk element ze horen.

Reflectie tot nu toe
Tot nu toe vind ik het erg fijn om met SASS te werken aangezien het een uitbreiding is op normaal CSS en een aantal handige onderdelen toevoegd die het schrijven van CSS makkelijker en overzichtenlijker maken. Daarnaast was de basis van SASS makkelijk te leren. Ik heb nu de basis van SASS onder de knie en ik ben van plan om ook in de toekomst SASS verder te blijven gebruiken bij het maken van deze website en nog meer functionaliteit ervan toe te passen. Daarnaast denk ik dat ik vanaf nu SASS ook standaard wil gaan gebruiken ald onderdeel van mijn workflow als het gaat om het ontwikkelen van websites of online applicaties nu ik geleerd heb hoe handig de functies ervan kunnen zijn.

Opbouw Portfoliowebsite

V

Ontwikkelomgeving | Beheerbaarheid

Inleiding
Onder dit kopje wil ik gaan beschrijven hoe ik mijn portfoliowebsite heb gemaakt. Ik wil dit kopje dus tijdens het semester blijven updaten met nieuwe dingen die ik heb gemaakt voor mijn website. Voor nu beschrijf ik hier de eerste opzet, de GitHub repository en mijn manier van werken met html, scss en javascript.

Opzet website
Aangezien ik voor semester 4 opnieuw een digitaal Portfolio ging maken en ik daarnaast ook nog een portfoliowebsite voor stage wilde gaan maken heb ik besloten om een eigen domein te laten hosten. Ik heb het domein RobinGalema.nl gekocht en hiervoor hosting aangeschaft.

Vervolgens heb ik een repository op GitHub aangemaakt waarmee ik het versiebeheer van mijn website ga bijhouden. Voor het ontwikkelen van mijn website heb ik ervoor gekozen om deze zelf te maken in plaats van een cms zoals WordPress of Webflow te gebruiken omdat ik het leuk vind om zelf mijn site te maken en graag complete vrijheid over de look van de site wil hebben. Daarnaast kan ik dan ook nog oefenen met het zelf maken van een website en het gebruik van html, css en javascript.

Het maken van de website doe ik met Visual Studio Code en voor dit semester heb ik besloten om te gaan werken met SASS waarover meer staat beschreven in het kopje hieboven. Na het installeren van de SASS live compiler en het aanzetten van mijn Live Server ben ik begonnen met het maken van mijn website.

Website tot nu toe
Tot nu toe heb ik de basis voor de website gemaakt met voor alle vakken een aparte pagina en een algemene landing pagina. De opdrachten staan momenteel gewoon als tekst in mijn html bestanden. In de toekomst wil ik kijken of ik deze content op een andere manier kan inladen zodat ik niet altijd mijn html hoef te bewerken om een aanpassing in de tekst te maken (met json bijvoorbeeld).
Daarnaast gebruik ik nu een beetje javascript om bijvoorbeeld op de landing-pagina naar beneden te scrollen naar de vakken en om het burger-menu te tonen en te verbergen.

Bij het schrijven van mijn code probeer ik over eenzeldfde notatie te gebruiken zodat al mijn code overzichtenlijk blijft. Ook geef ik elke functie een beschrijving met Jsdoc. Mijn eigen geschreven code is te lezen door op dit linkje te klikken.

Als voorbeeld staat hieronder een snippet uit mijn code waarin de functionaliteit van het openen en sluiten van het burgermenu geregeld wordt.

Gebruik van Git

Updated

V

Ontwikkelomgeving | Beheerbaarheid

Inleiding
Voor mijn portfoliowebsite en voor de proftaak gebruik ik GitHub voor mijn dev werk te beheren. Voor mijn website zal ik de laatste nieuwe versies bijhouden op GitHub zodat ik altijd een up-to-date versie online heb staan voor het geval er iets mis mocht gaan met de bestanden op de server of met mijn lokale bestanden. Voor de proftaak zullen we git hier natuurlijk ook voor gebruiken en daarnaast zullen we ook gebruik gaan maken van branches om samen aan ons project te werken en onderdelen van het product samen te ontwikkelen en deze delen vervolgens samen te voegen.

Git tot nu toe
Op dit moment ben ik alleen met mijn portfoliowebsite actief met Git bezig. Ik probeer elke keer dat ik een aantal aanpassingen doe deze in een commit te zetten en deze vervolgens naar een DEV branch te pushen zodat ik deze veilig online heb staan. Als een deel van de website af is en functioneert zoals het bedoeld is, merge ik dit stuk met master zodat hierin altijd de meest recente, werkende versie van mijn website staat.

Samenwerkingsafspraken NEW
Aan het begin van de eerste srpint hebben we een aantal afspraken gemaakt met de progtaakgroep over het gebruik van git en het schrijven van code zodat we hiermee allemaal op een lijn zitten. We hebben afgesproken om een repository te gebruiken voor al onze prototypes en deze te verdelen over individuele branches. Verder maken we voor elk prototype nog extra branches wanneer er meerdere mensen een prototype wordt gewerkt om conflicten te voorkomen. Doordat iedereen zijn eigen GitHub account gebruikt komt er een goed inzhicht wie waar heeft bijgedragen aan de hand van de commits.

Verder hebben we afgesproken om bij het maken functies altijd gebruik te maken van Jsdoc om de functie te beschrijven en dat we deze beschrijving altijd in het engels schrijven. Ook namen van variabele en functies zullen allemaal in het engels geschreven worden waarbij gebruikt wordt gemaakt van camelcasing. Op deze manier willen we onze code zo consequent mogelijk te schrijven en ervoor te zorgen dat iedereen binnen de groep altijd een makkelijk overzicht van functies en hun functionaliteit heeft.

Architectuur

Inleiding
Aan het einde van Gate 2 heb ik voor ons nieuwe concept een eerste Architectuurdiagram gemaakt. In dit diagram heb ik geprobeerd alle technische onderdelen te koppelen aan elkaar en na te denken over hoe onze uiteindelijke applicatie eruit komt te zien.

Het diagram
Voor het maken van het diagram heb ik eerst alle onderdelen van het museum die hiervoor relevant zijn opgeschreven. Dit zijn de volgende onderdelen:

  • Het hologram
  • De interactieve kaart
  • Een video-display
  • De mini-game
  • De lamp met afstandsensor

Vervolgens ben ik op draw.io begonnen met deze onderdelen aan elkaar te linken en na te denken welke andere onderdelen er nog nodig zijn om de applicatie werkend te maken.
Uiteindelijk heb ik het volgende diagram gemaakt voor ons mini-museum.

Voor het maken van dit diagram ben ik er van uit gegaan dat het hele museum vanuit een centrale applicatie draait. Dit willen we graag aangezien we op deze manier alle onderdelen tegelijk kunnen besturen. Hierdoor kunnen we als er niemand in de buurt is bijvoorbeeld alle video's stop zetten en schermen en lampjes uitzetten om stroom te besparen en vervolgens deze allemaal weer aan te zetten als er iemand in de buurt van het museum komt. Ook kunnen we zo onderdelen op elkaar afstemmen. Denk bijvoorbeeld aan het hologram dat veranderd op basis van het geselecteerde bedrijf op onze Brainport kaart.

Reflectie
Het maken van het diagram is vind ik goed gegaan, dit ook omdat we vooraf een aantal onderdelen van de installatie goed met de groep hebben besproken. Vervolgens heb ik redelijk eenvoudig een diagram in elkaar kunnen zetten voor onze applicatie. Toen we zijn begonnen met de eerste sprint ben ik met een deel van het diagram aan de slag gegaan. Aan alle onderdelen van het diagram kunnen we in de sprints user stories koppelen op het moment dat we aan deze gaan werken.

Zo heb ik in de eerste sprint aan de lamp gewerkt waarbij de volgende user story hoorde: "Als student wil ik dat het museum op mijn aanwezigheid reageert en daardoor mijn aandacht trekt." en waarbij weer een aantal kleinere tasks hoorden. Deze heb ik vervolgens met het architectuurdiagram in mijn achterhoofd kunnen uitvoeren.
Door een architectuurdiagram vooraf op te stellen heb ik tijdens het uitwerken van de losse onderdelen een beter beeld gekregen over hoe ik dit onderdeel in elkaar kon zetten en hoefde ik niet meer na te denken over hoe ik mijn prototype hoefde op te bouwen aangezien ik dit al bij het diagram heb gedaan. Dit is erg handig en hierdoor kan ik meteen aan de slag met het maken van de prototypes.

Voor het mini-museum willen we een lamp plaatsen die aangaat op basis van de aanwezigheid van een persoon rondom het museum. Als deze persoon binnen een bepaalde afstand van het museum komt moet deze lamp aangaan. Daarnaast moet de intensiteit van de lamp bepaald worden door de afstand van de dichtsbijzijnde persoon.

Om hier een simpel proof of concept van te maken ben ik aan de slag gegaan met een Arduino, een SR04 afstandsensor en de Johnny Five npm package.

Reflectie
Bij het maken van deze POC heb ik vooral een deel van mjn javascript kennis toegepast en herhaald. Het maken van deze POC ging opzich goed alleen duurde het in het begin lang om de goede firmata op mijn Arduino te krijgen doordat mijn MacBook constant geen permissie gaf om deze te installeren. Na even googlen heb ik dit op kunnen lossen en ben ik tijdens het maken van de code eigenlijk geen problemen tegengekomen. Ik volgens de afspraak binnen onze groep gewerkt aan de code, dat betekend dat ik Jsdoc heb gebruikt om functies te beschrijven en variabele in het engels en met camelcasing geschreven heb.

Hiervoor moeten we nog gaan zoeken naar een betere oplossing.
